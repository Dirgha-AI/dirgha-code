# Dirgha CLI 360° Audit Report

**Date:** 2026-04-23 05:02 UTC  
**Auditor:** Dirgha Code  
**Scope:** Whole-stack audit of Dirgha CLI with emphasis on bugs, logic errors, AI narration, streaming, TUI behavior, tool calls, memory, orchestration, long-horizon work, testing posture, and parity gaps vs top coding CLIs.

---

## Executive Summary

Dirgha CLI has substantial breadth and real engineering depth: multi-provider routing, tool calling, TUI streaming, narration, memory layers, orchestration, browser automation, fleet/worktree features, checkpointing, billing, and security controls. It is already beyond a prototype.

But it is **not yet at 100% parity with top-tier coding CLIs**. The current stack shows a recurring pattern:

- strong ambition and many subsystems
- partial implementation across several surfaces
- good mitigation layers for known failures
- but inconsistent contracts, placeholder/stub modules, high `any` / `@ts-nocheck` usage, and weak end-to-end guarantees across narration + streaming + long-horizon execution

### Current audit verdict

- **Product maturity:** advanced beta / pre-parity
- **Core agent loop:** workable, but still expensive and fragile in long sessions
- **Narration + streaming UX:** significantly improved, but still inconsistent across TUI, REPL, providers, and sub-agents
- **Tooling stack:** broad, but some tools are mismatched or semantically inconsistent
- **Memory + multi-agent:** powerful on paper, not fully integrated in a trustworthy way
- **Testing posture:** large suite exists, but real blind spots remain; one verified regression is present now

## P0 / Critical Findings

## 1. `read_file` contract regression

**Severity:** P0  
**Area:** tool calling, context control, downstream reasoning

### What is wrong
`read_file` is expected to return inline truncated content for large files. Instead, it is being post-processed by generic large-response offloading in `src/tools/index.ts`.

### Why it matters
A coding agent asks for `read_file` because it wants the file content in-context. Returning a pointer to a temp file is a different semantic contract.

### Evidence
- `src/tools/file.ts` already truncates/slices content
- `src/tools/index.ts` applies `offloadIfLarge()` to all tool results
- active failing test confirms mismatch

### Impact
- breaks tests
- can break model reasoning quality
- increases ambiguity in downstream orchestration and narration

### Recommendation
Make `read_file` a special-case:
- preserve inline truncated response contract
- avoid generic offload for `read_file`
- keep generic offload for search/web/shell outputs

---

## 2. Hot-context token bloat still exists in the main loop

**Severity:** P0  
**Area:** long-horizon work, cost, responsiveness

### What is wrong
The main agent loop still accumulates assistant outputs and tool-result blocks in `history`, and re-sends them across turns until compaction kicks in.

### Evidence
In `src/agent/loop.ts`:
- assistant response appended to `history`
- tool result blocks appended to `history`
- compaction only triggered after thresholds (`history.length > 60 || countTokensInMessages(history) > 80_000`)

### Why it matters
This is the core reason long-running tasks can still overpay. The leak is mitigated, not eliminated.

### Impact
- expensive long-horizon sessions
- degraded tool iteration speed
- weaker parity vs top coding CLIs that aggressively compress hot context

### Recommendation
Store two representations:
- **UI/raw transcript**
- **model-visible compact transcript**

Never keep raw large tool outputs in the model-visible transcript after first use.

---

## 3. Sub-agent / spawned-agent loops repeat the same history-retention pattern

**Severity:** P0  
**Area:** spawn_agent, orchestration, cost discipline

### What is wrong
`src/agent/spawn-agent.ts` also pushes assistant outputs and tool results into sub-agent history.

### Why it matters
Even if the parent only gets a summary back, the child execution itself can still be token-inefficient and unstable on long tasks.

### Impact
- orchestration cost multiplication
- deep task decomposition becomes expensive quickly
- parallel agents duplicate context burn

### Recommendation
Sub-agents should operate with:
- compact internal transcript
- explicit result schema
- bounded tool-result retention
- shared task brief instead of repeated full context

---

## 4. Memory stack is fragmented and not fully trustworthy

**Severity:** P0  
**Area:** memory correctness, context fidelity

### What is wrong
Dirgha CLI has multiple memory paths:
- builtin memory
- holographic memory
- unified memory
- command-layer remember/recall
- utility-layer unified-memory wrapper

But these do not form one clean, trusted contract.

### Evidence
- `src/utils/unified-memory.ts` is mostly stubbed:
  - `store()` empty
  - `retrieve()` returns null
  - `search()` returns []
  - `getContext()` returns []
- `src/memory/unified.ts` is `@ts-nocheck`
- memory tool names differ conceptually (`memory_store`, `remember`, `save_memory`, `read_memory`, etc.)

### Why it matters
A memory system that is partially symbolic, partially real, and partially stubbed creates false confidence. For a coding CLI, this is worse than having less memory.

### Recommendation
Unify memory around one canonical contract:
- persistent memory
- working/session memory
- semantic recall
- explicit tool naming

Deprecate or clearly separate overlapping paths.

---

## 5. Type safety debt is large enough to hide production bugs

**Severity:** P0  
**Area:** whole-stack maintainability, reliability

### What is wrong
The codebase contains extensive:
- `// @ts-nocheck`
- `as any`
- placeholder classes / no-op branches
- dynamic shape assumptions around provider messages, tool results, and UI events

### Evidence
Search results show widespread usage across:
- agent orchestration
- commands
- mesh
- voice
- project-session
- memory
- runtime
- TUI components

### Why it matters
This directly affects parity. Top CLIs feel reliable because contracts hold under stress. Here, many contracts are “best effort.”

### Recommendation
Create a strict hardening plan:
1. lock the agent/tool/provider message schema
2. eliminate `@ts-nocheck` from core loop, tool execution, memory, TUI streaming, providers
3. add invariant tests for every message conversion path

## P1 / High Priority Findings

## 6. Narration exists now, but classification and intent mapping are incomplete

**Severity:** P1  
**Area:** narration UX

### What is good
`src/agent/narration.ts` now exists and includes:
- tool classification
- intent rendering
- tool begin/end rendering
- agent begin/end rendering
- findings/phase/final rendering

This is real progress.

### What is still wrong
Narration quality is still uneven:
- some mappings are generic or misleading
- memory classification is inconsistent (`execute_sandbox` listed though real tool is `execute_code` elsewhere)
- user-facing intent strings are often mechanical rather than outcome-oriented
- narration is not uniformly applied across all execution surfaces (TUI, REPL, slash, spawned flows)

### Recommendation
Treat narration as a product system, not just a helper file:
- central event schema
- event source tagging
- compact, outcome-oriented phrasing
- consistent rendering across TUI and headless REPL

---

## 7. Parallel tool execution is correct for throughput but weak for coherent UX and safety

**Severity:** P1  
**Area:** tool calling, narration, determinism

### What is wrong
`executeAllTools()` uses `Promise.all(...)` over tool blocks.

### Why it matters
That means:
- narration can feel chaotic
- multiple heavy tools may run at once without a coherent progress story
- write-lock protection exists for file paths, but overall user experience is still bursty
- parallel reads/searches may drown the stream with non-chronological updates

### Recommendation
Use bounded and narratable concurrency:
- sequential by default for user-facing tool chains
- parallel only for explicitly independent classes (search, web, read-only)
- stable ordering in emitted narration events

---

## 8. Streaming stack is fragmented between REPL and TUI

**Severity:** P1  
**Area:** streaming parity, UX consistency

### What is wrong
There are two different high-level streaming experiences:
- legacy REPL path via `src/repl/index.ts` + `renderer.ts`
- Ink TUI path via `src/tui/App.tsx` + `StreamContainer`

These paths do not feel like one rendering architecture. They solve similar problems differently.

### Evidence
- REPL uses `StreamingRenderer`, `Spinner`, `renderToolBox`
- TUI uses stream events, buffered flush refs, event lists, tool stream events, status events
- status/thinking/tool rendering semantics differ

### Impact
- inconsistent bug surface
- duplicate fixes required
- parity drift between interfaces

### Recommendation
Converge on one event protocol and one rendering contract.

---

## 9. Thinking stream is surfaced, but not semantically robust yet

**Severity:** P1  
**Area:** reasoning UX, streaming

### What is wrong
The stack supports thinking deltas, but current handling is still shallow:
- some places count thinking chars instead of rendering meaningful semantic chunks
- provider-specific reasoning wrappers (`<think>...</think>`, `reasoning_content`, Anthropic thinking deltas) are normalized only partially
- some UIs reduce thinking to “chars received,” which is diagnostic, not premium UX

### Recommendation
Normalize all reasoning streams into one internal structure:
- `thinking_start`
- `thinking_delta`
- `thinking_end`
- visibility policy based on verbosity mode

---

## 10. Streaming resume logic is smart, but risky

**Severity:** P1  
**Area:** transport robustness, text correctness

### What is good
`src/providers/dispatch.ts` contains resume stitching logic for partial streamed text after connection drops.

### Risk
This is powerful, but string-based dedupe/stitching can still:
- duplicate fragments
- clip fragments
- create stitched text that no longer corresponds to provider truth

### Recommendation
Add transport-level replay tests with deterministic fixtures:
- interruption mid-sentence
- interruption during tool-call preamble
- interruption during reasoning tokens
- interruption before final tool block

---

## 11. SSE parsing is intentionally permissive, which can hide provider protocol drift

**Severity:** P1  
**Area:** provider compatibility

### What is wrong
`src/providers/http.ts` ignores malformed SSE lines.

### Why it matters
Silent parsing tolerance is good for resilience, but bad for auditability. If a provider changes format, the CLI may degrade silently instead of surfacing a clear transport warning.

### Recommendation
Keep permissive mode in prod, but add:
- debug counters for malformed SSE events
- structured warnings in verbose mode
- protocol conformance tests per provider adapter

---

## 12. Main loop file is still too large and too central

**Severity:** P1  
**Area:** maintainability, correctness

### What is wrong
`src/agent/loop.ts` remains the central gravity well for:
- billing
- context setup
- memory prefetch
- compaction trigger
- provider fallback
- reflection
- narration callback wiring
- tool execution
- stop reason handling

Even if line count has improved from older versions, it is still a risk concentration point.

### Recommendation
Split into explicit stages:
- setup_context
- request_model_turn
- process_model_response
- execute_tool_batch
- compact_if_needed
- finalize_turn

## P2 / Medium Findings

## 13. TUI has heavy complexity and multiple input/raw-mode paths

### What is wrong
There are several components dealing with raw mode and stdin event handling:
- `InputHooks.ts`
- `useKeyInput.ts`
- `HistorySearch.tsx`
- other overlays and pickers

This kind of TUI architecture often causes:
- prompt disappearance
- focus confusion
- stuck raw mode
- escape key inconsistencies

### Recommendation
Centralize raw-mode ownership.
Only one layer should own raw mode.

## 14. StreamContainer currently reduces thinking to counters in default rendering

### What is wrong
In `StreamContainer.tsx`, thinking is aggregated as character count.

### Why it matters
This is acceptable for diagnostics but not premium UX parity. Claude Code-level experience is about calm, legible, low-noise incremental state, not just counters.

## Narration-Specific Audit

## What is good
- narration module now exists
- tool classification and phase labeling exist
- loop wires `onFinding`, `onPhase`, `onAgentStart`, `onAgentEnd`
- TUI has status/tool/thought/text streams

## What is not yet parity-grade
- narration intent strings are still largely imperative (“Reading...”, “Writing...”) rather than decision-rich
- narration is not unified across REPL, TUI, slash commands, sub-agents, orchestration, and headless flows
- final synthesis quality is not consistently separated from intermediate status noise
- tool streams can still feel mechanically verbose rather than elegantly compressed

### Recommendation
Create a canonical narration event model:
- `phase`
- `decision`
- `action`
- `finding`
- `result`
- `error`

Then render that differently per mode:
- TUI compact cards
- REPL terse lines
- JSON structured events

## Streaming / TUI / Prompt Disappearance Audit

## Observed strengths
- explicit effort has gone into jitter reduction
- newline-gated streaming with hysteresis is present in TUI comments/logic
- tool_start/tool_stream/tool_end event protocol exists
- thinking stream support exists
- queue-based task processing exists

## Risks still present
- multiple input/raw-mode handlers
- dual rendering architectures (REPL + TUI)
- timer-heavy buffering logic across text/thinking/tool events
- high event complexity in `App.tsx`
- legacy + new streaming logic coexisting

### Likely symptoms that can still appear
- prompt visually lagging or disappearing during rapid tool updates
- transient stale tool boxes from previous turn
- thinking/status flicker when models emit mixed token kinds rapidly
- non-deterministic scroll behavior under dense event streams

### Recommendation
Add a streaming torture harness:
- 1000 incremental text chunks
- interleaved tool start/end
- simulated thinking stream
- delayed tool stream
- aborted turn mid-chunk
- resumed connection

Measure:
- render stability
- input focus integrity
- scroll persistence
- prompt visibility

## Claude Code / Top CLI Parity Gaps

To compete seriously with Claude Code-class UX, Dirgha CLI still needs:

1. **Single source of truth for streaming events**
2. **No visible contract drift in core tools**
3. **Aggressive hot-context compression by default**
4. **Stronger type guarantees across provider/tool/message boundaries**
5. **Fewer experimental surfaces in the default product story**
6. **Cleaner narration that explains intent without noise**
7. **Deterministic long-horizon behavior under stress**
8. **Scenario-driven parity tests, not only unit coverage**

## Recommended Action Plan

## Phase 0 — Immediate blockers (P0)

1. Fix `read_file` regression
2. Separate UI transcript from model-visible transcript
3. Compress tool results before re-inserting into model history
4. Audit and unify memory contracts
5. Start removing `@ts-nocheck` from core agent/tool/provider/TUI path

## Phase 1 — Streaming and narration hardening

1. Unify REPL + TUI event model
2. Normalize thinking/reasoning stream model
3. Add event-order tests for narration and stream rendering
4. Add stream recovery fixtures for provider interruptions
5. Centralize raw mode ownership

## Phase 2 — Multi-agent and orchestration hardening

1. Give spawned agents compact transcripts and strict result schemas
2. Reduce duplicated context across sibling agents
3. Bound parallel tool and agent concurrency by default
4. Consolidate multi-agent user-facing concepts

## Phase 3 — Parity-grade verification

1. Add full-stack scenario tests
2. Add long-horizon cost regression tests
3. Add TUI stability harness
4. Add provider protocol conformance tests
5. Only then update parity claims in docs

---

## Bottom Line

Dirgha CLI is **ambitious, real, and already impressive**. It has enough architecture to become a top-tier coding CLI.

But today’s evidence does **not** support a claim of 100% parity with Claude Code-class tools.

The core blockers are:
- contract inconsistency in core tools
- incomplete memory integration
- hot-context token bloat still present
- fragmented streaming/rendering architecture
- too much type suppression and placeholder logic in important surfaces

If these are fixed systematically, Dirgha CLI can absolutely compete.
If not, the product risks feeling powerful-but-fragile instead of premium and trustworthy.
