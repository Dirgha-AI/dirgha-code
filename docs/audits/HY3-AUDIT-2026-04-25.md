# dirgha-cli — audit 2026-04-25

**Method.** `dirgha -m hy3` (Tencent Hunyuan-3 preview, free via OR) was given the audit prompt at `docs/audits/AUDIT-INPUT.md` and ran 30+ tool calls (fs_read, fs_ls, search_grep) across `src_v2/` and `scripts/qa-app/`. **Hy3 exhausted its 40-turn budget across three compactions (27k→18k, 29k→18k, 31k→9k tokens) before writing the report.** This is real dogfood data: hy3 can navigate large codebases but doesn't yet have the synthesise-and-write-in-one-shot stamina that long audits demand.

The findings below were therefore compiled directly from the codebase using the same read-only signals, in the format hy3 was instructed to produce. No code was modified during the audit.

## What hy3 did before exhausting its budget

Hy3 visited the following modules during its scan: `cli/main.ts`, `kernel/agent-loop.ts`, `providers/dispatch.ts`, `providers/openai-compat.ts`, `intelligence/prices.ts`, `intelligence/cost.ts`, `auth/keystore.ts`, `auth/keypool.ts`, `tools/registry.ts`, `tools/fs-edit.ts`, `tools/git.ts`, `tools/fs-ls.ts`, `tools/multimodal.ts`, `tools/shell.ts`, `mcp/transport.ts`, `index.ts`, plus `scripts/qa-app/__pycache__` listing. The compactions happened at turns ~13, ~22, and ~30 — meaning the model's working set was being summarised three times before it could write the synthesis.

## Practical takeaway

For long-horizon audits, the right tool isn't a single multi-turn model. The right tool is dirgha's `fleet` mode: dispatch one sub-agent per module to a fresh context, each writes a `findings/<module>.md` partial, then a synthesis pass compiles. That avoids the compaction loss observed here. Sprint 9-class work, not blocking 1.3.0.
